# Imbalance-Aware-Crowd-Anomaly-Detection
Spatial Temporal Learning and Circulative Conditional GAN-Based Imbalance-Aware Crowd Anomaly Detection
## Section 1: Problem Statement Formulation
Rapid urbanization and wider deployment of smart surveillance have increased the need for automated monitoring in high-traffic locations such as transit hubs, stadiums, shopping centers, and busy intersections. Prompt detection of abnormal events—like panic-induced running, stampedes, violent incidents, or sudden crowd dispersals—is critical for public safety. Continuous, large-scale video feeds are impractical and error-prone for human operators, which has driven the development of automated crowd-anomaly detection systems capable of real-time monitoring.

Despite advances in deep neural networks, these systems face important limitations. A primary challenge is class imbalance: abnormal events occur far less frequently than normal crowd activities, and publicly available surveillance datasets are heavily skewed toward normal samples. This imbalance biases models toward routine behavior and reduces their ability to detect rare anomalies. Moreover, many existing approaches emphasize either spatial feature extraction or temporal sequence modeling, which restricts their capacity to fully capture the spatiotemporal dynamics of crowd behavior.
As such we brainstormed and realized an imbalance-sensitive crowd anomaly detection method which unites spatial feature learning,with generative data augmentation in one architecture.To solve the problem of class imbalance, the suggested approach uses a CGAN with circulative learning mechanism. Realistic anomaly-like samples are generated by the circulative method by iteratively refining the generator and discriminator. This generative augmentation approach makes the training data more diverse and enhances the ability of the model to detect infrequent anomalous patterns in crowded domains.

## WGAN-GP Loss Formulation:

### 1. Background: Problem with Traditional GANs

Traditional GANs use Jensen-Shannon (JS) divergence or KL divergence, which suffer from:
- **Vanishing gradients** when the discriminator becomes too good
- **Mode collapse** (generator produces limited varieties)
- **Unstable training** due to saturating loss functions

The JS divergence between two distributions that don't overlap is **constant** (log 2), providing no gradient signal.

### 2. Wasserstein Distance (Earth Mover's Distance)

**Definition:**
$$W(P_r, P_g) = \inf_{\gamma \in \Pi(P_r, P_g)} \mathbb{E}_{(x,y) \sim \gamma} [\|x - y\|]$$

Where $\Pi(P_r, P_g)$ is the set of all joint distributions whose marginals are $P_r$ (real data) and $P_g$ (generated data).

**Intuitive Explanation:**
Think of $P_r$ and $P_g$ as two piles of dirt. Wasserstein distance = **minimum work needed** to move one pile to match the other, where:
- Work = mass × distance moved
- Unlike JS divergence, this provides **meaningful gradients** even when distributions don't overlap

### 3. Kantorovich-Rubinstein Duality (Practical Computation)

We can't compute the infimum directly. The dual formulation gives:

$$W(P_r, P_g) = \sup_{\|f\|_L \leq 1} \mathbb{E}_{x \sim P_r}[f(x)] - \mathbb{E}_{x \sim P_g}[f(x)]$$

Where $f$ is a **1-Lipschitz function**.

**In GAN terms:** Let the critic $D(x)$ (formerly discriminator) approximate this optimal $f$:

$$W(P_r, P_g) \approx \max_{D \in 1\text{-Lipschitz}} \mathbb{E}_{x \sim P_r}[D(x)] - \mathbb{E}_{z \sim P_z}[D(G(z))]$$

### 4. Critic (Discriminator) Loss
![1](https://github.com/lavyagaur/Imbalance-Aware-Crowd-Anomaly-Detection/blob/main/Images/Criticloss.png)
### 5. Generator Loss

$$\mathcal{L}_G = -\mathbb{E}_{z \sim P_z}[D(G(z))]$$

**Why negative?** 
- The generator wants to **maximize** the critic's score on fake samples (to trick the critic)
- Minimizing $-\mathbb{E}[D(G(z))]$ is equivalent to maximizing $\mathbb{E}[D(G(z))]$

### 6. Gradient Penalty (GP) - The Core Innovation

$$\text{GP} = \mathbb{E}_{\hat{x} \sim P_{\hat{x}}}\left[(\|\nabla_{\hat{x}} D(\hat{x})\|_2 - 1)^2\right]$$

**Where $\hat{x}$ comes from:**
$$\hat{x} = \epsilon x_{real} + (1-\epsilon) x_{fake}, \quad \epsilon \sim U[0,1]$$

**Purpose:** Enforce the 1-Lipschitz constraint softly via penalty.

### 7. Essence of 1-Lipschitz Constraint

#### What is Lipschitz Continuity?

A function $f$ is **K-Lipschitz** if:
$$\|f(x) - f(y)\| \leq K \cdot \|x - y\| \quad \forall x, y$$

For WGAN, we need **1-Lipschitz** ($K=1$): the function cannot change faster than its input changes.

**Gradient norm constraint:** $\|\nabla_x D(x)\|_2 \leq 1$ almost everywhere.

#### Why Without the Constraint?

**Problem 1: Unbounded Gradients**
Without Lipschitz constraint:
D(x) can have arbitrarily steep slopes
→ Discriminator can perfectly separate real/fake with infinite margin
→ Wasserstein estimate → ∞ (useless)
→ Vanishing gradients for generator

**Problem 2: No Valid Wasserstein Estimate**
The Kantorovich-Rubinstein duality **only holds** for 1-Lipschitz functions. Without it, you're not estimating Wasserstein distance at all.


#### Why With the Constraint?

**Benefit 1: Smooth Gradient Landscape**
- Critic's output changes gradually as samples move
- Provides meaningful gradients even when distributions are far apart
- No saturating regions

**Benefit 2: Valid Wasserstein Approximation**
- The critic's loss directly approximates Earth Mover's Distance
- Lower Wasserstein distance = more similar distributions
- Provides a **training monitoring metric** (lower is better, not fooled by overfitting)

**Benefit 3: Improved Stability**

| Without Constraint | With 1-Lipschitz |
|-------------------|------------------|
| Exploding gradients | Controlled gradients |
| Mode collapse common | Diverse outputs |
| Sensitive to hyperparameters | More robust |
| Vanishing gradients | Consistent signal |

### 8. Comparison: Weight Clipping (Original WGAN) vs Gradient Penalty

**Original WGAN (weight clipping):** Force weights $w$ to $[-c, c]$
- Problem: Often pushes weights to extremes (-c or c)
- Problem: Too restrictive (capacity problem)
- Problem: Ignores function's actual gradient norm

**WGAN-GP (gradient penalty):** Directly enforce $\|\nabla D(\hat{x})\|_2 = 1$
- Cleaner constraint
- Allows richer function class
- Better stability and convergence

## Section 3: Aggressive Metastability Control

### 3.1 The Core Problem: Discriminator Overpowering

In WGAN-GP training, a common failure mode occurs when the discriminator (critic) learns too quickly relative to the generator:

```
Healthy Training:           Unstable Training:
    D loss ≈ 2-8               D loss > 15
    WD ≈ 0.5-5                 WD > 10
    ↔ Balanced                  → D overpowering
```

**Symptoms of D Overpowering:**
- |D_loss| grows rapidly (>15) → Critic confidently separates real/fake
- |Wasserstein distance| spikes (>10) → Distributions diverging
- Generator loss stagnates → No useful gradient signal
- Mode collapse begins → Generator produces limited varieties

### 3.2 Controller State Machine
**Three States:**

| State | Trigger Condition | Meaning |
|-------|-------------------|---------|
| **NORMAL** | \|D_loss\| ≤ 15 AND \|WD\| ≤ 10 | Training stable, no intervention needed |
| **D_OVERPOWER** | \|D_loss\| > 15 OR \|WD\| > 10 | D gaining advantage, moderate intervention |
| **SEVERE_D_OVERPOWER** | \|D_loss\| > 25 OR \|WD\| > 30 | D severely overpowering, emergency measures |

**State Transitions:**
```
NORMAL ──(instability detected)──→ D_OVERPOWER
D_OVERPOWER ──(severe threshold)──→ SEVERE_D_OVERPOWER
D_OVERPOWER ──(recovery)──→ NORMAL (counter decrements to 0)
SEVERE_D_OVERPOWER ──(recovery)──→ NORMAL (after actions applied)
```

### 3.3 Instability Counter

Rather than triggering on a single bad epoch, the controller maintains a persistent counter:

```
instability_count += 1    when instability detected
instability_count -= 0.5  when epoch is healthy (slower recovery than detection)
instability_count = max(0, instability_count)  // never negative
```

**Design rationale:**
- **Fast reaction:** Detects instability immediately (count ≥ 1)
- **Slow recovery:** Prevents oscillation (takes 2 healthy epochs to recover from 1 bad)
- **Short memory:** 3-epoch sliding window for loss history

### 3.4 Control Actions

#### Action 1: Gradient Damping

When triggered, discriminator gradients are scaled down:

```
∇_θD ← γ · ∇_θD    where γ = 0.2
```
**Effect:** D learns 5× slower this batch, giving G time to catch up.

#### Action 2: Generator Gradient Boost

During D overpower, generator gradients are amplified:

```
∇_θG ← β · ∇_θG    where β = 2.0
```

**Effect:** G takes bigger steps toward fooling D, counteracting D's advantage.

#### Action 3: Learning Rate Modulation

| State | α_D (D factor) | α_G (G factor) | Effective η_D | Effective η_G |
|-------|----------------|----------------|---------------|---------------|
| Normal | 1.0 | 1.0 | 0.0001 | 0.0002 |
| D_OVERPOWER | 0.3 | 1.5 | 0.00003 | 0.0003 |
| Initial Handicap | 0.5 | 1.0 | 0.00005 | 0.0002 |

```
η_D = α_D × η_D_base    (base = 0.0001)
η_G = α_G × η_G_base    (base = 0.0002)
```

**Effect during overpower:** D learns at 30% speed, G learns at 150% speed → G closes the gap.

#### Action 4: Update Ratio Control

```
n_critic_dynamic = max(0, n_critic_phase - 1)
```

When `skip_d_update` is triggered:
```
n_critic_dynamic = 0    // D gets NO updates this batch
```

**Effect:** G gets a "free" generation step without D improving, allowing it to produce better samples.

### 3.5 Intervention Levels

The controller uses |WD| as a continuous health metric:

```
|WD| < 0.01     → 🔴 MODE COLLAPSE (D cannot distinguish real from fake)
0.01 ≤ |WD| ≤ 10 → 🟢 HEALTHY (competitive equilibrium)

```

### 3.8 Why This Approach Works

| Problem | Without Controller | With Aggressive Controller |
|---------|-------------------|---------------------------|
| D learns too fast | D dominates within 5 epochs | D handicapped, G catches up |
| Mode collapse | G produces 1-2 patterns | Diverse outputs maintained |
| Gradient vanishing | No signal to G | G boosted, D damped |
| Training divergence | Unrecoverable crash | Early detection + intervention |
| Oscillation | Alternating D/G wins | Counter-based persistence |

### 3.9 Limitations

- **Heuristic nature:** Thresholds (15, 10, 25) are empirically chosen, not theoretically derived
- **No theoretical guarantees:** Does not provably prevent mode collapse
- **May over-correct:** Aggressive damping could slow convergence if triggered unnecessarily
- **Dataset-specific:** Optimal thresholds may vary across datasets
- **Counter design:** Asymmetric increment/decrement (1.0 vs 0.5) is a design choice

## Section 4: Data Pipeline: Construction, Annotation, and Contamination

### 4.1 Dataset Overview

The dataset is procedurally generated synthetic imagery designed to simulate "motion energy" patterns:

```
Total frames:     500
├── Anomaly:       25  (5.0%)   — Frame indices 0-24
└── Normal:       475  (95.0%)  — Frame indices 25-499

Image dimensions:  64×64×3 (RGB)
Format:            JPEG files saved to disk
```

### 4.2 Anomaly Class Generation (High Motion Energy)

Each anomaly frame (indices 0-24) is created with chaotic, high-entropy patterns:

**Algorithm:**
```
1. Initialize black canvas (64×64×3, all zeros)
2. Draw n_streaks ~ UniformInt(8, 15) random directional streaks:
   - Start position: (x, y) ~ UniformInt(0, 60)²
   - Length:         L ~ UniformInt(15, 40)
   - Thickness:      T ~ UniformInt(1, 3)
   - Direction:      Uniform from {horizontal, vertical, diagonal_up, diagonal_down}
   - Color:          R ~ UniformInt(200, 255), G,B ~ UniformInt(50, 100)
3. Add sparse noise: Bernoulli(p=0.2) per pixel → bright colors
4. Apply GaussianBlur(kernel=3×3) for motion blur effect
5. Label: "anomaly" (binary 1)
```
**Visual characteristics:** Motion streaks, directional blur, chaotic high-frequency noise, reddish warm colors suggesting rapid movement.

### 4.3 Normal Class Generation (Low Motion Energy)

Each normal frame (indices 25-499) is created with structured, organized patterns:

**Algorithm:**
```
1. Fill canvas with background: UniformInt(80, 160) for all RGB channels
2. Add vertical structures every 6 columns (width 2px):
   frame[:, col:col+2, :] ~ UniformInt(100, 140)
3. Add horizontal structures every 8 rows (height 2px):
   frame[row:row+2, :, :] ~ UniformInt(90, 130)
4. Add block-wise variations in 16×16 blocks:
   color_shift ~ UniformInt(-10, +10) per channel
   Clamp to [80, 160]
5. Label: "normal" (binary 0)
```

### 4.4 Annotation Mechanism

**Critical Design Choice:** Labels are assigned by **frame index position**, not by content analysis:

```
IF frame_index < 25:
    label = "anomaly"
ELSE:
    label = "normal"
```

**Implications:**

| Aspect | Detail |
|--------|--------|
| Annotation method | Position-based, not content-based |
| Temporal signature | First 5% of sequence always anomalous |
| Content independence | Label does not depend on visual features |
| Randomness effect | Each run generates different images with same labels |

**Risk:** A model could learn to detect "first 25 frames" rather than actual anomaly patterns, especially if frame order information leaks through data loading.

### 4.5 Class Imbalance and Oversampling

**Original Class Distribution:**

```
Normal:   475 (95%)  ████████████████████████████████████████████████
Anomaly:   25 (5%)  ███
```

**Oversampling Procedure:**

```
Target anomaly ratio:  ρ = 0.3 (30%)
n_anomaly_target      = floor(475 × 0.3) = 142
n_to_generate         = 142 - 25 = 117 additional copies

Method: Bootstrap resampling with replacement
Source: 25 original anomaly frames
Copies: 117 drawn randomly from the 25 originals
Expected copies per original: 117/25 ≈ 4.68
```

**Post-Oversampling Distribution:**

```
Normal:   475 (77.0%)  ████████████████████████████████████████
Anomaly:  142 (23.0%)  ████████████
Total:    617 frames
```

**Code Implementation:**
```python
def oversample_minority_class(df, target_ratio=0.3):
    normal_df = df[df['label'] == 'normal']      # 475 samples
    anomaly_df = df[df['label'] == 'anomaly']     # 25 samples
    
    n_anomaly_target = int(len(normal_df) * target_ratio)  # 142
    n_to_generate = n_anomaly_target - len(anomaly_df)      # 117
    
    anomaly_oversampled = resample(
        anomaly_df,
        replace=True,           # WITH replacement
        n_samples=n_to_generate
    )
    
    balanced_df = pd.concat([df, anomaly_oversampled])
    return balanced_df
```

### 4.7 Data Leakage Analysis

**The Contamination Path:**

```
Step 1: 25 original anomaly frames (A₀...A₂₄) generated
Step 2: All 25 go to TRAINING set (indices 0-24 < 431)
Step 3: 117 COPIES of A₀...A₂₄ created via bootstrap
Step 4: All 117 copies go to TEST set (indices 500-616 > 431)
Step 5: Test set contains EXACT duplicates of training patterns
```

### 4.8 Types of Leakage

| Leakage Type | Description | Severity |
|-------------|-------------|----------|
| **Exact Duplicate** | Pixel-identical copies in train and test | 🔴 Critical |
| **Pattern Overlap** | Same 25 scenes in both sets | 🔴 Critical |
| **Temporal Position** | Anomalies always first 25 frames | 🟠 High |
| **No Scene Separation** | No mechanism to isolate novel scenes | 🟠 High |

### 4.9 Impact on Evaluation

**Without Leakage (Proper Setup):**
```
Train: Scenes 1-17     → Generator learns these patterns
Test:  Scenes 18-25    → Generator tested on NEW patterns
                         Score reflects TRUE generalization
```

**With Leakage (Current Setup):**
```
Train: Scenes 1-25     → Generator learns ALL patterns
Test:  Scenes 1-25     → Generator tested on KNOWN patterns
       (copies)          Score reflects MEMORIZATION
```

**Effect on Metrics:**

| Metric | Without Leakage | With Leakage | Bias |
|--------|----------------|--------------|------|

| Precision | True detection rate | Inflated | Over-optimistic |

## Section 5: Anomaly Detection Protocol

### 5.1 Core Principle: Reconstruction-Based Detection

The system uses the **AnoGAN principle**: a generator trained on both normal and anomaly patterns will reconstruct familiar patterns better than unfamiliar ones.

```
Assumption:
  Generator learns P(x|y) for both y=0 (normal) and y=1 (anomaly)
  
  BUT: Normal patterns are structured, organized, lower-entropy
       → Easier to learn → Better reconstructions → Lower MSE
  
  Anomaly patterns are chaotic, high-entropy, streak-heavy
       → Harder to learn → Worse reconstructions → Higher MSE
```

**Note on Leakage:** Because test anomalies are copies of training anomalies (Section 4.7), the generator has already seen all anomaly patterns. The score separation relies on **reconstruction difficulty** (entropy difference), not **novelty detection**.

---

### 5.2 Anomaly Score Computation

**Algorithm:**

```
For each test image (x, y):
    
    errors = []
    
    For k = 1 to 3:
        z_k ~ N(0, I₁₀₀)                    // Sample random latent vector
        x̂_k = G(z_k, y)                     // Generate reconstruction
        e_k = MSE(x, x̂_k)                   // Pixel-wise error
        errors.append(e_k)
    
    anomaly_score = mean(errors)            // Average over 3 attempts
```

**Mathematical Formulation:**

```
s(x, y) = (1/K) · Σ(k=1 to K) MSE(x, G(z_k, y))

where:
  K = 3                    (number of reconstruction attempts)
  z_k ~ N(0, I₁₀₀)        (independent random latent vectors)
  MSE(a, b) = (1/CHW) · Σ(a - b)²
            = (1/(3×64×64)) · Σ_{c,h,w} (a_{c,h,w} - b_{c,h,w})²
```

**Code Implementation:**

```python
def detect_anomalies(self, test_loader, n_samples=3):
    self.generator.eval()
    all_scores = []
    all_labels = []
    
    with torch.no_grad():
        for imgs, labels, _ in test_loader:
            for i in range(imgs.shape[0]):
                img = imgs[i:i+1]              # Single image
                label = labels[i:i+1]
                
                recon_errors = []
                for _ in range(n_samples):
                    z = torch.randn(1, 100)    # Random noise
                    gen_img = self.generator(z, label)
                    error = torch.mean((img - gen_img) ** 2)
                    recon_errors.append(error.item())
                
                anomaly_score = np.mean(recon_errors)
                all_scores.append(anomaly_score)
                all_labels.append(label.item())
    
    return self._calculate_metrics(all_scores, all_labels)
```

### 5.3 Why K=3 Samples?

**Statistical Justification:**

```
Single sample:     score = MSE(x, G(z₁, y))
                   Variance: σ²

Three samples:     score = (1/3) · [MSE(z₁) + MSE(z₂) + MSE(z₃)]
                   Variance: σ²/3

Reduction:         3× lower variance → more stable scores
```

**Computational Trade-off:**

| K | Variance Reduction | Inference Time | Marginal Benefit |
|---|-------------------|----------------|------------------|
| 1 | Baseline (σ²) | 1× | — |
| 2 | σ²/2 | 2× | Large |
| 3 | σ²/3 | 3× | Moderate |
| 5 | σ²/5 | 5× | Small |
| 10 | σ²/10 | 10× | Minimal |

K=3 balances stability with computational cost for CPU-optimized inference.

### 5.5 Threshold Selection Protocol

**Algorithm:**

```
Given: Scores S = {s₁, s₂, ..., s₁₈₆} from test set
       Labels Y = {y₁, y₂, ..., y₁₈₆} (ground truth)

Step 1: Generate precision-recall curve
  For each unique score value τ (descending):
    pred_i(τ) = 1 if s_i > τ else 0
    
    TP(τ) = Σ(pred_i=1 AND y_i=1)
    FP(τ) = Σ(pred_i=1 AND y_i=0)
    FN(τ) = Σ(pred_i=0 AND y_i=1)
    
    Precision(τ) = TP / (TP + FP)
    Recall(τ)    = TP / (TP + FN)

Step 2: Calculate F1 for each threshold
    F1(τ) = 2 · P(τ) · R(τ) / (P(τ) + R(τ) + ε)

Step 3: Select optimal threshold
    τ* = argmax F1(τ)
           τ

Step 4: Apply threshold
    predictions = (S > τ*).astype(int)
```

## Section 7: Reproducibility Analysis

### 7.1 Definition of Reproducibility

A computational experiment is **reproducible** if another researcher, using the same code and data, can obtain **identical** results. This requires:

```
1. Fixed random seeds (eliminates stochastic variation)
2. Deterministic algorithms (no non-deterministic operations)
3. Identical software environment (same library versions)
4. Same hardware or controlled numerical precision
```

**Different from replicability:** Obtaining consistent conclusions with different code/data.

---

### 7.2 Current Reproducibility Status

The code has **no reproducibility mechanisms**. Running it twice produces different results every time.

**Root Causes:**

```python
# Cause 1: No random seed set anywhere
import numpy as np
# np.random.seed() is NEVER called

# Cause 2: PyTorch default random initialization
torch.nn.Linear(...)  # Weights randomly initialized each run

# Cause 3: Training noise sampling
z = torch.randn(batch_size, 100)  # Different z every run

# Cause 4: Bootstrap without fixed state
resample(anomaly_df, replace=True, n_samples=117)  # No random_state
```

---

### 7.3 Component-by-Component Reproducibility Assessment

#### 7.3.1 Fully Reproducible Components (Deterministic)

These produce identical output given identical input:

| Component | Why Reproducible |
|-----------|-----------------|
| **Model Architecture** | Fixed layers, fixed dimensions. `CPUOptimizedGenerator` and `EmergencyDiscriminator` always create the same structure. |
| **Loss Functions** | WGAN-GP loss is pure math: `-E[D(real)] + E[D(fake)] + λ·E[(‖∇D‖-1)²]` |
| **Gradient Penalty** | Deterministic given `x_real`, `x_fake`, `α`. Interpolation and gradient computation have no randomness. |
| **Training Phase Schedule** | Hardcoded epoch boundaries: e<5 → EMERGENCY, 5≤e<10 → TRANSITION, e≥10 → NORMAL |
| **Metastability Thresholds** | Hardcoded constants: 15.0, 10.0, 25.0, 30.0 |
| **Optimizer Configuration** | Fixed hyperparameters: Adam, lr values, β=(0.5, 0.999) |
| **Data Split Point** | Index-based split at `int(617 × 0.7) = 431`. Same every run. |
| **Batch Size** | Fixed at 16 |
| **Early Stopping Logic** | Deterministic rule: if `consecutive_unhealthy ≥ patience`, stop |
| **LR Modulation Formula** | `ηD = αD × 0.0001`, `ηG = αG × 0.0002`. Deterministic given α values. |

---

#### 7.3.2 Partially Reproducible Components (Same Algorithm, Different Output)

These follow the same procedure but produce different results due to unseeded randomness:

| Component | What's Deterministic | What's Random |
|-----------|---------------------|---------------|
| **Dataset Generation** | Algorithm (streaks, grids, colors), label assignment by index | Exact pixel values (positions, lengths, colors drawn from unseeded distributions) |
| **Oversampling** | Target count (117 copies), method (bootstrap with replacement) | Which 25 originals get copied and how many times each |
| **DataLoader Sampling** | WeightedRandomSampler logic | Which specific samples appear in each batch |

**Example — Dataset Generation:**
```python
# Same algorithm every run:
num_streaks = np.random.randint(8, 15)    # Run 1: 11, Run 2: 9, Run 3: 13
start_x = np.random.randint(0, 60)        # Run 1: 23, Run 2: 47, Run 3: 5
direction = np.random.choice(['H','V','D']) # Run 1: 'V', Run 2: 'H', Run 3: 'D'

# Each run generates DIFFERENT images
# But the LABEL (anomaly/normal) is the same by index
```

---

#### 7.3.3 Not Reproducible Components (Stochastic)

These vary every execution with no mechanism to control them:

| Component | Source of Randomness | Impact |
|-----------|---------------------|--------|
| **Weight Initialization** | PyTorch default `kaiming_uniform_` | Different starting point → different convergence path |
| **Training Noise (z)** | `torch.randn(B, 100)` each batch | Different fake samples each step |
| **Interpolation (α)** | `torch.rand(B, 1, 1, 1)` each GP | Different interpolation points |
| **Training Trajectory** | Combined effect of all above | Different loss curves, WD values, convergence speed |
| **Wasserstein Distance** | Depends on model state | Different every epoch, every run |
| **Early Stopping Epoch** | Depends on WD trajectory | May stop at epoch 15 in run 1, epoch 42 in run 2 |
| **Anomaly Scores** | 3 random z per test image | Different scores each inference |
| **Optimal Threshold τ*** | Depends on anomaly scores | Different τ* each run |
| **Final Metrics (P, R, F1)** | Depends on τ* and scores | Different reported performance each run |

## Section 8: Contribution to Real-Time Surveillance

### 8.1 Overview

This system is a **proof-of-concept** for anomaly detection in crowd monitoring scenarios. While it operates on synthetic static images without temporal processing, the underlying approach has a pathway to real-time surveillance deployment.

**Current State vs. Deployment Goal:**

```
Current:  Synthetic 64×64 images → Offline processing → Frame-level texture classification
Goal:     Live video feed → Real-time processing → Temporal anomaly detection with alerts
```

---

### 8.2 Theoretical Contribution Pathway

#### 8.2.1 Frame-Level Anomaly Scoring

The core mechanism applies directly to surveillance:

```
For each video frame x_t at time t:

    s_t = (1/K) · Σ(k=1 to K) MSE(x_t, G(z_k, y=0))
    
    where:
    - x_t is the current frame from CCTV feed
    - G is trained on normal crowd patterns (y=0)
    - s_t measures deviation from "normal" appearance
```

**Interpretation in surveillance context:**

| Score Range | Meaning | Example Scenario |
|-------------|---------|------------------|
| s_t < 0.03 | Normal crowd | People walking calmly |
| 0.03 ≤ s_t < 0.08 | Mild deviation | Crowd density increasing |
| 0.08 ≤ s_t < 0.15 | Suspicious | Sudden movement, running |
| s_t ≥ 0.15 | Anomaly | Panic, stampede, fight |
